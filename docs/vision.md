# Boost Searcher 愿景

> 让任何 C++ 工程师都能在几秒内定位到可靠、上下文完整的 Boost 文档。

## 为什么要做

- **社区痛点**：Reddit 讨论《Is it just me, or is the quality of the Boost API documentation...》集中吐槽断链、版本混乱、缺少示例。
- **团队需求**：在企业内训或新人 onboarding 时，Boost 文档往往成为效率黑洞。
- **个人使命**：延续 2023 年在 CSDN 完成的原型，让项目成为一份可放进 GitHub/简历的“文档修复典范”。

## 目标用户画像

| 角色 | 诉求 |
| --- | --- |
| C++ 工程师（初中级） | 快速确认“此函数在哪个头文件/命名空间？有没有示例？” |
| 资深/架构师 | 检索历史版本差异、定位设计讨论/动机文档。 |
| 技术写作者 | 依据搜索引擎数据补写官方文档、提交 PR。 |
| 社区贡献者 | 反馈断链、贡献镜像链接或示例。 |

## 核心场景

1. **断链修复**：用户点击搜索结果时，若目标 404 或跳转老版本，界面提示“该链接不可用/建议镜像”。
2. **摘要速览**：在搜索列表即显示关键词高亮、相关度评分、所属组件，让用户无需打开多个标签。
3. **结构化字段**：提供头文件、命名空间、示例链接、相关版本等信息，弥补官方文档缺失。
4. **贡献流程**：在 UI / README / docs 中明确“如何反馈”“如何提交 PR”“AI 写作模板”。

## 问题列表（持续更新）

- [ ] Parser 无法区分链接是否失效，需要添加 HTTP 检测。
- [ ] 正排数据缺少 `header`, `namespace`, `example_url` 等字段。
- [ ] 前端缺少版本筛选、断链提示、可访问性模式。
- [ ] 文档尚未形成完备的贡献指南与路线图展示。

## 成功标准

- 用户搜索常见 Boost 模块时，能在首页看到可用链接 + 关键上下文。
- 断链提示命中率 ≥ 90%，并可选择镜像/跳转方式。
- README + docs 让新贡献者在 10 分钟内了解项目背景与启动方法。
- Roadmap 与 Issue/Discussion 保持同步，形成持续迭代节奏。
